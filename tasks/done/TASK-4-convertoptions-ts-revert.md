# TASK-4: Move DNxHR doc comment from generated TS file to Rust source

## Goal
The hand-edit on `src/lib/types/generated/ConvertOptions.ts` is reverted. The DNxHR resolution constraint documentation lives on the Rust `dnxhr_profile` field as a `///` doc comment, where ts-rs can propagate it. The next `cargo build` regenerates the TypeScript file with the doc comment intact instead of silently overwriting a hand-edit.

## Context
The working tree currently has an uncommitted modification to a ts-rs-generated file:

```
M src/lib/types/generated/ConvertOptions.ts
```

The diff inserts a JSDoc block mid-struct, at the `dnxhr_profile` field:

```typescript
/**
 * Requires output resolution ≥ 1280×720; returns error if opts.resolution is set below this.
 */
dnxhr_profile: string | null,
```

The file's first line says: `// This file was generated by [ts-rs](https://github.com/Aleph-Alpha/ts-rs). Do not edit this file manually.`

The annotation content is correct and useful — it captures the BC-005 encoder-constraint rule for DNxHR. The location is wrong: the next `cargo build` (with ts-rs export enabled) regenerates this file from the Rust struct and silently drops the comment.

The fix is to put the doc comment on the Rust struct field. ts-rs supports propagating Rust doc comments to TypeScript via `///` comments out of the box (ts-rs ≥ 7.0).

Relevant files:
- `src/lib/types/generated/ConvertOptions.ts` — revert the hand-edit
- `src-tauri/src/lib.rs` — find the `ConvertOptions` struct, locate `pub dnxhr_profile: Option<String>`, add the `///` doc comment above it
- `src-tauri/Cargo.toml` — verify ts-rs version supports doc propagation

## In scope
- Revert `src/lib/types/generated/ConvertOptions.ts` to its committed state.
- Add `/// Requires output resolution ≥ 1280×720; returns error if opts.resolution is set below this.` immediately above the `dnxhr_profile` field declaration in the Rust `ConvertOptions` struct.
- Run `cargo build` (or `cargo test`) to re-trigger ts-rs codegen and verify the doc comment appears in the regenerated TS file.

## Out of scope
- Other field doc comments — only `dnxhr_profile` was hand-edited in the working tree.
- Migrating other ad-hoc constraint docs (e.g. AVIF speed cap) to `///` comments. Could be a follow-up sweep.
- Bumping ts-rs version.
- Refactoring the ConvertOptions struct.

## Steps
1. Verify the diff: `git diff src/lib/types/generated/ConvertOptions.ts` shows only the JSDoc insertion.
2. Find the `ConvertOptions` struct in `src-tauri/src/lib.rs`. It's a `#[derive(...)]`'d struct with `#[ts(export)]` and many `Option<...>` fields. `dnxhr_profile` should be near the video-codec fields.
3. Add the doc comment directly above the field:
   ```rust
   /// Requires output resolution ≥ 1280×720; returns error if opts.resolution is set below this.
   pub dnxhr_profile: Option<String>,
   ```
4. Revert the TS file: `git checkout HEAD -- src/lib/types/generated/ConvertOptions.ts`
5. Trigger ts-rs codegen — ts-rs runs as part of `cargo test` typically (export tests). Run:
   ```
   cargo test --manifest-path src-tauri/Cargo.toml --lib export_bindings
   ```
   If no export-test target exists, fall back to `cargo build` and check for the regenerated file.
6. Diff the regenerated `ConvertOptions.ts` against HEAD. Expected delta: one inserted JSDoc block above `dnxhr_profile`. If the regen produced no doc comment, ts-rs in this version doesn't propagate `///` automatically — switch to the explicit attribute:
   ```rust
   #[doc = "Requires output resolution ≥ 1280×720; returns error if opts.resolution is set below this."]
   pub dnxhr_profile: Option<String>,
   ```
7. `cargo fmt --manifest-path src-tauri/Cargo.toml`

## Success signal
- `git diff src/lib/types/generated/ConvertOptions.ts` shows the same JSDoc block is now produced by codegen (or a similar comment in ts-rs's preferred format).
- `git diff src-tauri/src/lib.rs` shows the new `///` (or `#[doc]`) line on `dnxhr_profile`.
- The codegen-produced file is no longer a hand-edit — re-running `cargo build` produces no diff against the committed version.
- `cargo build --manifest-path src-tauri/Cargo.toml` exits 0.

## Notes
- This is a 5-minute fix. The risk is silent: if the hand-edit is committed and a colleague runs `cargo build` later, their diff will include "ConvertOptions.ts modified" with the comment removed, triggering needless review back-and-forth.
- If ts-rs doesn't propagate the doc in this project's version, the fallback `#[doc = "..."]` form virtually always works. If neither works, the doc belongs in a separate `KNOWN-CONSTRAINTS.md` doc, with a Rust-side comment pointing at it. Generated files must remain pristine.
- The constraint itself is already enforced in `convert/video.rs` per BC-005 / commit `0d1c045`. This task only addresses where the documentation lives.
